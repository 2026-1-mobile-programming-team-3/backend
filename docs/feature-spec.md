총 **44개 기능**을 6개 카테고리로 구분한다. 우선순위 정의는 `프로젝트정보.md` 8절을 참고.

> 학교 프로젝트 시연 범위에 맞춰 운영성 기능(차단)은 제외하고 신고·역할 전환은 최소 형태로 간소화함. DB 구조는 `db/DB설계.md` 참고.

## 1. 사용자 관리 (Auth) — 14개

### 1.1 회원가입 [T0 / 하 / 백엔드]
- 사용자에게 이메일, 비밀번호, 닉네임, (선택) 연락처를 받아 회원 등록을 한다.
- 비밀번호는 8자 이상 영문+숫자+특수문자를 포함해야 한다.
- 정상 처리 시 `201 CREATED`, 검증 실패 시 `400`, 이메일 중복 시 `409`.

### 1.2 로그인 [T0 / 하 / 백엔드]
- 이메일/비밀번호를 검증하고 JWT Access Token + Refresh Token을 발급한다.
- Access Token 만료: 30분, Refresh Token 만료: 7일.

### 1.3 토큰 갱신 [T0 / 하 / 백엔드]
- Refresh Token을 사용하여 새 Access Token을 발급한다.
- 만료된 Refresh Token은 재로그인을 요구한다.

### 1.4 로그아웃 [T0 / 하 / 백엔드]
- 현재 사용 중인 Refresh Token을 블랙리스트에 등록하여 무효화한다.
- 클라이언트에서는 로컬에 저장된 토큰을 함께 삭제한다.

### 1.5 내 정보 조회 [T0 / 하 / 백엔드]
- 로그인한 사용자 본인의 프로필 정보를 조회한다.
- 등록된 반려동물 목록도 함께 반환한다.

### 1.6 계정 정보 수정 [T0 / 하 / 백엔드]
- 닉네임, 연락처, 프로필 이미지 등 사용자 정보를 부분 수정(Partial Update)한다.
- 변경할 필드만 전송 가능.

### 1.7 비밀번호 변경 [T1 / 하 / 백엔드]
- 현재 비밀번호를 검증한 뒤 새 비밀번호로 변경한다.
- 새 비밀번호는 회원가입 시와 동일한 유효성 규칙을 적용한다.

### 1.8 계정 탈퇴 [T1 / 하 / 백엔드]
- Soft delete 방식으로 사용자 계정을 비활성화한다.
- 탈퇴 시 비밀번호 재확인 필요. 30일 후 영구 삭제.

### 1.9 반려동물 등록 [T0 / 하 / 백엔드]
- 사용자의 반려동물 정보(이름, 종류, 품종, 나이, 체중, 중성화 여부, 사진)를 등록한다.
- 한 사용자가 여러 마리 등록 가능.

### 1.10 반려동물 수정 [T1 / 하 / 백엔드]
- 본인이 등록한 반려동물의 정보를 수정한다.
- 본인이 아닌 경우 `403`, 존재하지 않는 ID는 `404`.

### 1.11 반려동물 삭제 [T1 / 하 / 백엔드]
- 등록된 반려동물 정보를 삭제한다 (`204 No Content`).

### 1.12 봉사자 역할 전환 요청 [T1 / 중 / 백엔드]
- 일반 사용자(USER)가 봉사자(VOLUNTEER) 역할 전환을 요청한다.
- 신청 메시지(자유 텍스트) 한 줄만 제출한다.
- 동일 사용자가 동시에 PENDING 요청 2개 이상 보낼 수 없다 (`409`).
- 관리자 승인 후 역할이 변경되며, 결과는 알림으로 통보된다.

### 1.13 (관리자) 봉사자 요청 목록 조회 [T1 / 중 / 백엔드]
- 관리자가 봉사자 역할 전환 요청 목록을 조회한다.
- 상태별(PENDING/APPROVED/REJECTED) 필터링 및 페이지네이션 지원.

### 1.14 (관리자) 봉사자 요청 승인/거부 [T1 / 중 / 백엔드]
- 관리자가 요청을 승인 또는 거부한다 (action 만 전송).
- 승인 시 사용자 role이 `VOLUNTEER`로 변경되고 알림이 발송된다.
- 이미 처리된 요청은 `409`.

## 2. 알림 (Notification) — 4개

### 2.1 알림 목록 조회 [T0 / 하 / 백엔드]
- 로그인한 사용자의 알림 목록을 최신순으로 조회한다.
- 읽음/안읽음 필터, 카테고리 필터(VOLUNTEER, MATCH, REVIEW, NEWS, POLICY, SYSTEM) 지원.
- 페이지네이션 지원.

### 2.2 알림 전체 읽음 처리 [T1 / 하 / 백엔드]
- 사용자의 모든 안읽은 알림을 일괄로 읽음 처리한다.
- 처리된 알림 개수를 응답한다.

### 2.3 읽지 않은 알림 개수 조회 [T0 / 중 / 백엔드]
- 앱 상단 배지(badge) 표시를 위한 가벼운 카운트 조회 API.
- 폴링 또는 앱 포그라운드 진입 시 호출.

### 2.4 푸시 알림 디바이스 토큰 등록 [T0 / 중 / 백엔드]
- FCM 디바이스 토큰을 서버에 등록한다.
- 한 사용자가 여러 기기를 사용할 수 있으므로 다중 토큰 관리.
- 현재 안드로이드만 지원.

## 3. 중성화 이동 지원 매칭 (Match) — 11개

> 본 카테고리는 원본 명세에서 상세 API 정의가 누락된 항목이 다수 있어, 기능 요구사항 위주로 정리한다. API 상세는 추후 확정 예정.

### 3.1 이동 지원 요청 글 작성 [T0 / 하 / 백엔드]
- 취약계층 보호자(또는 봉사자)가 중성화 수술 이동 지원이 필요한 경우 요청 글을 작성한다.
- 제목, 내용, 위치(좌표), 희망 일정, 반려동물 정보를 포함한다.

### 3.2 이동 지원 요청 글 수정/삭제 [T1 / 하 / 백엔드]
- 본인이 작성한 요청 글의 수정 및 삭제. 매칭이 진행되기 전까지만 가능.

### 3.3 이동 지원 요청 목록 조회(필터 포함) [T0 / 중 / 백엔드]
- 봉사자가 신청 가능한 요청 목록을 조회한다.
- 지역, 상태(WAITING/MATCHING/PROGRESS/DONE), 일정 등 필터 지원.

### 3.4 특정 요청 상세 정보 조회 [T0 / 하 / 백엔드]
- 요청 글의 상세 정보(반려동물, 위치, 일정, 신청자 수 등) 조회.

### 3.5 봉사 신청하기 [T0 / 중 / 백엔드]
- 봉사자가 특정 요청에 대해 봉사를 신청한다.
- 신청자 정보가 작성자에게 알림으로 전달된다.

### 3.6 신청자 목록 조회(작성자용) [T0 / 중 / 백엔드]
- 요청 작성자가 자신의 글에 신청한 봉사자 목록을 조회한다.
- 봉사자 프로필, 차량 보유 여부 등 매칭 판단 정보 포함.

### 3.7 봉사자 매칭 수락/거절 [T0 / 중 / 백엔드]
- 작성자가 신청자별 채팅으로 대화한 뒤 한 명을 수락하거나 거절한다.
- 거절은 채팅 진입 없이 신청자 카드에서도 즉시 가능.
- 수락 시 `matches.status` `MATCHING` → `PROGRESS`로 전이, 같은 매칭의 다른 PENDING 신청은 모두 자동 `REJECTED` + 양측 알림 발송.

### 3.8 매칭 상태 업데이트(진행/완료) [T1 / 중 / 백엔드]
- 매칭 진행 단계(WAITING → MATCHING → PROGRESS → DONE)를 갱신한다.
- HTTP `PATCH /matches/{id}/status` 호출로 단순 처리. 클라이언트는 화면 진입/새로고침 시 최신 상태 조회.
- (실시간 WebSocket은 데모 안정성 위해 채택하지 않음)

### 3.9 1:1 채팅 메시지 전송/조회 [T1 / 중 / 백엔드]
- **신청 발생 직후부터** 작성자가 신청자와 1:1 대화 가능 (수락 전 단계). 채팅으로 신뢰 형성 후 채팅방 안에서 매칭 결정.
- 채팅 스레드 = `match_application` 1건. 매칭 1건당 N개의 스레드(작성자 ↔ 각 신청자)가 동시에 존재.
- 거절된 application의 스레드는 비활성화(메시지는 보존). 수락된 application의 스레드는 PROGRESS 단계 일정 조율에 그대로 사용.
- 클라이언트는 채팅 화면 진입 시 메시지 목록 조회 + 주기적 폴링.

### 3.10 봉사 완료 인증 및 후기 작성 [T1 / 중 / 백엔드]
- 봉사 완료 후 사진/메모로 인증하고 상호 후기를 작성한다.
- 통계 누적의 기준이 된다.

### 3.11 개인별 누적 봉사 이력 통계 [T2 / 중 / 백엔드]
- 봉사자 개인별 누적 봉사 횟수, 시간, 평점 등 통계 제공.
- 마이페이지에 노출.

## 4. 신고 (Report) — 2개

> 차단 기능은 데모 범위 외로 제외. 신고는 단일 백엔드(`POST /reports`)로 통합.

### 4.1 게시글/후기 신고 등록 [T1 / 하 / 백엔드]
- 부적절한 게시글·후기 작성자를 신고한다.
- 신고 대상 사용자 ID와 사유(자유 텍스트)만 전송.
- 동일 신고자→대상 중복 신고는 차단한다 (`409`).

### 4.2 채팅 내 유저 신고 등록 [T1 / 하 / 백엔드]
- 채팅 중 부적절한 발언을 한 사용자를 신고한다.
- 백엔드는 4.1과 동일한 `POST /reports` 엔드포인트를 호출 (UI 진입점만 다름).

## 5. 지도 / 장소 서비스 (Map) — 9개

### 5.1 반려동물 출입 가능 매장 지도 조회 [T0 / 상 / 백엔드]
- 사용자 현재 위치(`lat`, `lng`) 기반 반경 내 출입 가능 매장의 좌표·기본 정보 조회.
- 기본 반경: 2,000m (변경 가능).
- 인증 없이 호출 가능.

### 5.2 주변 봉사(중성화 이동 지원) 위치 표시 [T0 / 중 / 백엔드]
- 봉사자에게만 표시되는 이동 지원 요청 위치 마커 조회.
- **봉사자 권한 필요** (일반 사용자 접근 시 `403`).

### 5.3 매장 상세 정보 조회 [T0 / 하 / 백엔드]
- 특정 매장 마커 클릭 시 운영시간, 사진, 상세 출입 조건, 평균 평점 등을 조회.

### 5.4 매장 검색 및 결과 조회 [T1 / 중 / 백엔드]
- 상호명 또는 주소 키워드로 매장을 검색한다.

### 5.5 지도 마커 필터링(카테고리, 출입조건) [T1 / 중 / 백엔드]
- 카테고리(CAFE/RESTAURANT/PARK), 반려동물 출입 가능 여부 등으로 마커를 필터링한다.

### 5.6 신규 매장 정보 등록 [T1 / 중 / 백엔드]
- 사용자가 새로운 반려동물 동반 가능 매장을 등록 요청한다.
- 등록 직후 상태는 `PENDING` (관리자 검수 후 노출).

### 5.7 매장 정보 수정 및 삭제 [T2 / 하 / 백엔드]
- 자신이 등록한 매장만 수정/삭제 가능. 관리자는 모든 매장 가능.

### 5.8 매장 리뷰 조회 [T1 / 하 / 백엔드]
- 특정 매장에 달린 리뷰 목록(별점, 닉네임, 반려동물 출입 가능 여부, 작성일) 조회.

### 5.9 매장 리뷰 작성 및 삭제 [T2 / 하 / 백엔드]
- 매장에 별점·리뷰·**반려동물 출입 가능 여부 체크**를 작성하거나, 본인이 작성한 리뷰를 삭제.
- 한 사용자가 같은 매장에 중복 리뷰 작성 불가 (409).
- 매장의 출입 가능 정보는 두 출처에서 나온다: 관리자 입력 (공식) + 사용자 리뷰 체크 (현장 확인).
